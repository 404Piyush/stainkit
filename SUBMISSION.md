## Code Project Description (paste into the form)

### Purpose

Digital pathology labs routinely need to normalise the colour
appearance of histopathology slides before feeding them into
machine-learning models. The two stains most commonly used — haematoxylin
(stains nuclei blue/purple) and eosin (stains cytoplasm pink) — are
applied inconsistently across labs, scanners, and even batches from
the same lab. A model trained on one lab's slides can fail outright on
another lab's slides for no reason other than colour.

This project implements **Macenko (2009) stain normalisation** and
**Ruifrok-Johnston (2001) colour deconvolution** on the GPU. The
pipeline takes a whole-slide image patch as input and produces:

1. A stain-normalised RGB image, where the same tissue looks the
   same regardless of which lab processed it.
2. A binary tissue mask that separates tissue from background/glass
   using Otsu thresholding + morphological opening + closing.

### Algorithms

The mathematical contracts are documented in `docs/algorithms.md`. In
summary:

* **Colour deconvolution.** Each pixel's (R, G, B) is converted to
  optical density (negative logarithm). The resulting OD vector is
  multiplied by the inverse of the 3x3 stain matrix to obtain per-pixel
  haematoxylin, eosin and residual concentrations.
* **Macenko stain-basis estimation.** Each pixel's (H, E) OD pair is
  projected onto the stain plane, converted to (angle, magnitude), and
  the 1st and 99th percentile angles are taken to be the directions
  of H and E. These are lifted into 3D unit vectors.
* **Macenko reconstruction.** The estimated H and E vectors are
  replaced with a *target* stain matrix and user-supplied
  concentrations, and the result is exponentiated back to RGB.
* **Otsu tissue masking.** The image is converted to luminance, a
  256-bin histogram is built on the device, and the threshold that
  maximises inter-class variance is computed. The resulting binary
  mask is post-processed with a morphological opening and closing of
  radius `params.otsu_smoothing_radius` to remove dust and fill
  small holes.

### CUDA implementation

The pipeline is built as three CMake libraries:

* `stainkit_core` — host-side image I/O, types, CPU reference
  implementation, and pipeline orchestration.
* `stainkit_cuda` — five CUDA kernels (color deconvolution, optical
  density conversion, stain-normalisation, Otsu tissue masking,
  separable binary morphology).
* `stainkit` — the CLI executable.

Key engineering decisions:

* **Cross-TU ABI safety.** After debugging a SIGSEGV traced to a
  `StainMatrix` ABI mismatch across `.cu` translation units on
  Colab's CUDA 12.8 runtime, all cross-`.cu` interfaces now take raw
  `float*` pointers instead of complex value types. Every kernel
  launcher takes `cudaStream_t` directly instead of the fragile
  `void*` pattern that was dereferencing parameter slots
  incorrectly.
* **Multi-stream-friendly.** The `Pipeline` class owns 8 CUDA streams
  and serialises per-image work through `cudaEvent_t`-based stage
  timing so the benchmark CSV can attribute wall time to specific
  stages (H2D copy, deconvolution, normalisation, mask, D2H copy).
* **SIGSEGV recovery.** `Pipeline::MakeOrFallback()` installs a
  SIGSEGV/SIGBUS handler around CUDA init so that an ABI mismatch
  with the host driver falls back to the CPU reference implementation
  instead of terminating the whole process.

### Results

Tested on Google Colab with a Tesla T4 (sm_75, CUDA 12.8):

| Image size | CPU (ms/img) | GPU (ms/img) | Speedup |
| ---------- | -----------: | -----------: | ------: |
| 256 × 256  |        6.05  |        5.22  |   1.16x |
| 512 × 512  |       27.06  |       19.35  |   1.40x |

On small (256²) patches the GPU and CPU are roughly comparable
because launch overhead and the H2D/D2H transfers dominate. The
colour-deconvolution kernel alone is **~10x faster on the GPU than
on the CPU**, and on larger images the relative advantage grows
because the kernel work scales with `npix` while launch overhead
stays constant.

### Lessons learned

1. **The `void* stream` parameter pattern is fragile.** Passing a
   `cudaStream_t` by value implicitly converts to `void*` but
   dereferencing the void* as `cudaStream_t*` to read the value is
   undefined behaviour unless the caller passes a `cudaStream_t*`
   explicitly. We hit this in two places — the `AsStream` helper in
   five `.cu` files, and the cross-`.cu` `StainMatrix` ABI mismatch
   that took the longest to track down because it only manifested on
   Colab's specific runtime. Lesson: take `cudaStream_t` directly in
   every kernel launcher.
2. **Stain-basis estimation collapses on uniform input.** The
   percentile-based Macenko algorithm needs at least two distinct
   stain populations in the input. The bundled synthetic image
   generator uses a single eosin background + single hematoxylin
   cell colour, which causes the angle distribution to be unimodal
   and the estimated basis to collapse to a single vector — visible
   in the docs/screenshots/ panels as a uniform purple normalised
   output. Real histopathology images (PatchCamelyon, TCGA) work
   correctly because they contain many shades of pink and purple.
3. **`cudaEvent_t` lifetime matters.** We initially read
   `ScopedEvent::elapsed_ms` inside the inner scope, before the
   destructor's stop-event synchronisation ran. All benchmark values
   were 0.00 ms. Hoisting each `ScopedEvent` to function scope and
   adding an explicit `MarkStop()` step fixed this.
